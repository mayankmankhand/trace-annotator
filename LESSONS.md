## What I Learned
<!-- Add entries after each session where you learned something new -->

- **XML tags in prompts are a real thing, not just hype.** All three major providers (Anthropic, OpenAI, Google) recommend XML for complex, multi-section prompts. The key insight: XML helps AI distinguish *what kind of content* a section is (rules vs templates vs examples), which markdown headers alone can't do.
- **Hybrid approach beats all-or-nothing.** Don't replace markdown with XML - use both. Markdown headers stay for human readability, XML wraps content blocks for AI parsing. Pattern: header outside the tag, content inside.
- **AI peer review recommendations often over-engineer.** When /ask-gpt and /ask-gemini reviewed the XML plan, they recommended namespaced tags (`tk_rules`), a formal schema file, testing rubrics, and a three-wave rollout. Most of that was unnecessary for 9 small files. Trust your gut when something feels like too much.
- **A worktree is just a folder, a branch is just a pointer.** They're independent. Deleting a worktree folder does not delete the branch, PR, or any commits. You can always re-create a worktree from the same branch. Think of it like closing a document window vs. deleting the file.
- **Worktree detection: `--git-dir` vs `--git-common-dir`.** To check if you're in a Git worktree, compare `git rev-parse --git-dir` with `git rev-parse --git-common-dir`. If they differ, you're in a worktree. The first attempt used `--show-toplevel` vs `--git-common-dir`, which compares a directory path to a `.git` path - they're never equal, so it always said "worktree." The review process caught this before it shipped.

- **Use deterministic scripts for structural data, not LLMs.** When generating a file tree index, the first instinct was to have Claude scan and describe every file. Both GPT and Gemini called this out: "using an LLM to generate a token-saving index burns more tokens than it saves." A 20-line Node script using `git ls-files` does the job in milliseconds with zero token cost. Reserve LLMs for judgment calls, not mechanical tasks.
- **Version bumps touch more files than you think.** Six files track the toolkit version: VERSION, package.json, package-lock.json, .claude/rules/toolkit.md (stamp comment), CHANGELOG.md, and AGENT-SETUP.md (title + "What's new" block). Easy to miss one - v4.2.1 shipped with package.json, package-lock.json, and toolkit.md stale; v4.1 missed toolkit.md too. **Use `bash scripts/setup/bump-version.sh <ver>`** - it handles the 4 mechanical files and prints a checklist for the 2 prose files. An earlier version of this lesson claimed "no automated fix possible because the checklist would leak to downstream users" - that was wrong. `scripts/setup/` is maintainer-only; setup.sh's propagation allowlist (setup.sh:352-354) only copies the three runtime scripts, so anything else added to `scripts/setup/` stays in the toolkit repo.
- **Never interpolate shell variables into inline `node -e` strings.** If `$TARGET` contains a single quote (e.g., `/home/user/it's-a-project`), it breaks the JS string literal and `2>/dev/null` silently swallows the error. Pass paths via environment variables instead: `TARGET_DIR="$TARGET" node -e "const dir = process.env.TARGET_DIR; ..."`. This avoids all quoting issues.
- **`settings.json` can get modified by research subagents.** During the index format research, a subagent fetching aider.chat added a WebFetch permission and additionalDirectories to `.claude/settings.json`. Always run `git diff` before committing to catch unintended changes from subagent side effects.
- **Permission approvals can land in the wrong settings file.** When Claude Code prompts for a new Bash permission and the user approves it, the entry can get appended to `.claude/settings.json` (shared/committed baseline) instead of `.claude/settings.local.json` (user-specific, where the project convention says real permissions live). Also captures literal argument values, so test-run entries like `Bash(bash scripts/setup/bump-version.sh not-a-version)` end up in the file. Mitigation: `git diff .claude/settings.json` after approving any new permission, and move the entry to `settings.local.json` with a broader glob (`Bash(bash scripts/setup/foo.sh *)`) if the script is meant to be allowed long-term.

- **"Do I understand it?" vs "Can I use it?" is the right split for review skills.** When adding `/review-copy`, the key insight was that clarity/orientation and usability/accessibility are genuinely different concerns. A page can be perfectly usable (good contrast, accessible forms) but still confuse a newcomer who doesn't know what they're looking at. Separate skills with a clear tie-break rule ("meaning/orientation = copy, interaction/accessibility = UX") prevents muddled feedback.
- **Run debates on plans, not just finished work.** Running `/ask-gpt` and `/ask-gemini` on the plan (before building) caught four refinements that would have required rework later: audience assumption wording, severity anchor calibration, auto-detection heuristics, and the need for a non-goals section. Cheaper to fix a plan than to rewrite a shipped skill.

## Mistakes to Avoid
<!-- Add patterns that caused problems so they don't repeat -->

- **Don't run `npm run build` while the dev server is running.** Turbopack's incremental compilation and the production-build outputs both write to `.next/`, and they corrupt each other's state. Symptom: dev server starts returning HTTP 500 with `MODULE_NOT_FOUND` and `ENOENT: ...build-manifest.json` errors that survive a refresh. Recovery is annoying: kill all `next dev` processes (`ps aux | grep "next dev"`), `rm -rf .next`, then restart. `npm run build` is a CI gate; do not run it on a workstation that has a dev server up. Hit this twice during v3 testing before remembering.
- **Don't micro-tag individual bullets.** Tag sections, not lines. Wrapping every bullet in XML adds noise without helping parsing.
- **Watch for tool output artifacts in reviews.** The Read tool's output wrapper (`</output>`) can look like actual file content. Always verify with raw bytes before "fixing" something that might not be broken.
- **Review your own AI-generated code before shipping.** The initial worktree detection logic had a fundamental bug that would have broken every worktree feature. Running `/review-commands` and `/review-code` caught it. The review workflow exists for a reason - don't skip it even when the code "looks right."
- **Skill tool expansions can serve stale command versions.** Root cause: old command files in `~/.claude/commands/` (global/home directory) override the project's `.claude/commands/` files. In our case, 9 files were manually copied to the home directory in January before setup.sh existed, then never updated. When slash commands ran, the AI used the global copies instead of the current project versions - missing Staff Engineer Check, Finding IDs, and other updates. Fix: delete global copies (commands should only live in each project folder) and setup.sh now warns if it detects conflicting global files. See issue #52.

- **AI debates surface things standard reviews miss.** Running `/ask-gpt` and `/ask-gemini` alongside the 5 `/review-*` commands found insights none of the reviews caught: tone/task decoupling (shift questions but keep tone after scope narrowing) and the need for a vision-mode closing deliverable template. The debates push harder on behavioral edge cases because two models are actively challenging each other.
- **"Same command, two gears" vs "new command" decision pattern.** When debating whether to add a mode or create a separate command, the key question is: do they share enough structure? If the two behaviors use the same phases, question format, and guardrails (just different content), one command with modes. If they need different structure or workflow, separate commands. Applied this to `/explore` vision mode - both gears share Phase 1/Phase 2 structure, just different questions and tone.
- **CLI default-acceptance prompts don't fully translate to chat.** A prompt like `Scoping or vision? [scoping]` assumes the user can accept the bracketed default by hitting enter. In chat surfaces (Claude Code, Cursor), users always type *something* - the "press enter" branch is mostly dead. Prompts that use this convention need (a) synonym parsing for natural-language answers ("let's go vision", `v`, "big picture"), and (b) an explicit acknowledgement when input is ambiguous, rather than silently falling back to the default. Caught during the issue 88 review before it shipped.
- **When changing user-facing copy, grep for the same description elsewhere.** Updating README.md's `/explore` row drifted away from the toolkit.md description of the same command - same command, two stories. `/review-copy` flagged it. Cross-document consistency is a real review category, not just a nice-to-have.

### Skills migration decisions (issue #71)

- **Subagents do NOT auto-discover project skills.** The orchestrator must read skill files and pass content explicitly. You cannot assume a subagent will "just find" a skill in `.claude/skills/`.
- **Cross-directory file references in skills use `` !`cat ...` `` dynamic injection syntax, not `!include`.** This is how shared content gets pulled into skill prompts at runtime.
- **`@axe-core/playwright` is compatible with `playwright-core`** (peer dependency). No conflict with the library-only Playwright install the toolkit uses.
- **`user-invocable: false` works as documented.** It hides the skill from the slash command menu, but Claude can still invoke it programmatically. Useful for agent-only skills like `project-context`.
- **Shared files reduce duplication across review skills.** `.claude/skills/shared/` eliminated roughly 48 lines of duplicated content across 6 review skills.
- **GPT and Gemini peer review caught things standard reviews missed.** Zombie process risk, setup script legacy cleanup gap, missing documentation files, and an axe-core compatibility question all came from the debate rounds, not from `/review-*` commands.

## Patterns That Work
<!-- Add approaches and conventions that proved effective -->

- **Tag vocabulary for prompts:** `<rules>` (must-follow), `<procedure>` (step-by-step), `<template>` (copy-paste schemas), `<output_format>` (expected output), `<conditions>` (if-then logic), `<guidelines>` (best practices), `<reference>` (lookup tables), `<phase>` (named stages), `<examples>` (good/bad demos).
- **Audit before converting.** Not every file needs XML. Short files, human-facing docs, and already-structured data are fine without it. Focus on files where the AI needs to distinguish content types.

### Rename-aware setup cleanup (issue #80)

- **`for i in "${!array[@]}"` is the idiomatic Bash way to iterate parallel indexed arrays.** First pass used `i=0; while [ $i -lt ${#arr[@]} ]; do ...; i=$((i+1)); done`. Both work and both are Bash 3.2 safe, but `${!array[@]}` expands to the index list, scopes the loop variable to the for-loop, and removes the manual counter. Reviewer caught it on the first pass. Use the for-form whenever you need indexes.
- **Scope parity gaps explicitly when fixing one of several.** While adding the rename block to setup.ps1, I noticed it was also missing the v3.4->v3.5 skills cleanup AND v4.0 plan migration that setup.sh has. Added only the rename block (issue #80 scope), surfaced the rest as a follow-up. Without explicit scoping, fixing "the obvious thing" tends to drag in adjacent fixes and turn a one-issue PR into a sprint.
- **A 🟩 on "ask the user about X" can mislead.** The plan-compliance reviewer flagged that marking the sub-task "ask whether to file a follow-up issue" as done was technically correct (I asked) but could read as "follow-up filed." For decision sub-tasks, either keep them 🟨 until acted on, or add a one-liner to Outcomes noting current status.

### browse.js hardening (issues #82 / #84 / #87)

- **Auto-start lifecycle code is hard to verify without a real dev server.** The fixes for #82 (`child.unref`, port snapshot, orphan-PID-don't-kill) all change `scripts/browse.js` process-management behavior, but exercising them requires `autoStart: true` plus simulated port conflicts and stale PIDs. Pipe-based smoke tests covered stdin parsing, browser launch, and `handleGoto` well; lifecycle changes shipped on code review only. Plan to do an end-to-end exercise on the next downstream `/review-browser` run that uses `autoStart: true`.
- **FS-probes for "is X installed" should fall through, not be authoritative.** First version of the Playwright cache-dir probe returned `BROWSER_NOT_FOUND` whenever the dir was missing - which broke Windows users (Playwright caches at `%LOCALAPPDATA%\ms-playwright`, not `~/.cache/ms-playwright`). Fix: only treat the probe as definitive when the dir EXISTS and is empty. Anything else falls through to the brittle-but-broader string match. The lesson generalizes: when you add a "smarter" check next to an existing fallback, gate the smart check on positive evidence, not on absence of evidence.
- **Lazy locators don't throw - validate at parse, not at construction.** `page.getByRole('buton')` does not throw; it returns a locator that finds nothing and times out 5s later. The original plan said "wrap getByRole in try/catch" - which would have caught nothing. The right fix was (a) validate the prefix syntax at parse time (`role:`, `role::Submit`, `role:button:` all throw immediately) and (b) thread the original target string into `ERR.TIMEOUT` so the eventual timeout names what the user tried. Pattern: when wrapping a deferred API, the wrapper has to either eagerly validate the input or carry context forward to the eventual error.

### Trace Annotator v2.0 launch

- **Never ship a "permanently dismiss" flag without a versioned key.** The v1 coaching cards never appeared for me because a stale `localStorage["ta:coaching:done"]=true` was set during dev and I forgot. Bumping the suffix to `:v2` was the cheap fix. Deeper lesson: anything stored in localStorage with permanent semantics needs a recovery path. Tying the flag to a per-file fingerprint (the v2 milestone cards) is better than tying it to "all sessions forever."
- **Silent persistence failures are the worst kind of bug.** v1 burned me twice: once with the coaching flag, once with an IndexedDB `try { load() } catch {}` that swallowed errors. If the catch falls through to "fresh start", the user can't tell whether they're truly fresh or whether their work is invisible. Always log on the catch and surface a flag the UI can render.
- **`<datalist>` is a silent UI when empty.** A native `<input list="...">` with zero options shows nothing - no dropdown, no placeholder, no cue. For "recognition over recall" UIs, use a visible chip strip plus an empty-state copy line, not just a datalist. Users opened it, saw nothing, didn't know what to do.
- **Reviewers find more issues than you should fix.** A 4-specialist `/review` produced 42 findings on v2.0. Triaging by *real-world impact for this user* (solo PM, local-only, learning) cut it to 5 actual fixes. Reviewers calibrate for enterprise-grade ship gates; you have to apply your own filter.
- **When broadening auto-recognition, re-check that "should fall through" fixtures still fall through.** Adding `submitted_text`/`model_text` as recognized field synonyms broke the deliberately-ambiguous test fixture - it stopped being ambiguous. The smoke test caught it. Lesson: synonyms expand both the success path and the "this should still hit manual mapping" path.

### Trace Annotator v3.0 (experienced mode + power features)

- **Auto-detection should walk one level into top-level objects.** The most common LLM eval file shape in the wild is OpenAI's request/response wrapping (`{request: {messages: [...]}, response: {messages: [...]}}`). Flat-field-only detection misses it - the wizard sees `request` and `response` as unfamiliar names and falls through to manual mapping, where the dropdown only shows top-level keys. A bounded depth-2 walk that prefers paths whose array contains an assistant turn gets the right answer without any typing. Generalizes: when auto-detection fails on real-world files, the right move is usually "look slightly deeper," not "give better manual mapping UX."
- **`<datalist>` is silent ENOUGH when populated too.** v2.0's lesson was "empty datalist shows nothing." v3 extended it: even populated, `<input list="...">` often does not pop a dropdown on click in some browsers - the user has to start typing. For "I don't know what to put here" UIs, render suggestions visibly above the input, not behind a native combobox. The "Detected paths" panel does this for the wizard.
- **Diagnostic errors are recoverable; generic errors are support tickets.** Replacing "got an unexpected value type" with "request is an object with keys: messages, try `request.messages`" turned the wizard from a dead end into a self-service path. Pattern: every error message should name (a) the actual shape of what was found, (b) the shape that was expected, (c) a concrete next step the user can act on without reading the source.
- **`setState` updaters must be pure - side effects belong outside.** React calls updater functions twice in dev StrictMode. Side effects inside (audit writes, undo-stack pushes, setState-on-other-state) double-fire. Pattern: read via `useStateRef.current`, compute the next state outside, call `setState(nextValue)` with a non-updater value, do side effects after. This came up across `applyVerdict`, `applyBatchVerdict`, `applyBatchTag`, `applyToolCallReview`, `toggleSkip`, `updateAnnotation` - all share the same shape after the fix.
- **`/review` finds blockers worth fixing IN the release, not after.** The unified review surfaced 2 hard UX blocks (no confirm on bulk overwrite, no way to multi-select) that would have shipped publicly. Folding them into the same release cost ~2-3 hours; shipping then patching in v3.0.1 would have cost an extra release cycle and a "yikes, that just overwrote my labels" Slack from a tester. Don't treat `/review` as the LAST gate before tagging - treat it as a budget check on whether to ship now or take one more day.
- **Vision-mode `/explore` converges fast when each round forces a hard pick.** 10 candidate v3 issues -> 5 in v3.0 + 2 in v3.1 + 5 cut, across two rounds. The forcing functions were "what does the LinkedIn comment say?" (audience clarity), "stay or graduate?" (story coherence), and the explicit Scope Dial (Expand / Hold / Reduce). Without those, vision conversations tend to keep adding rather than cutting.
- **Capture plan deviations in Outcomes BEFORE commit.** Three deviations landed during v3 execution (mode kept local in TraceView vs lifted to AppShell, no `saveLabelsBatch` helper, VERSION file not bumped because it holds the toolkit version). Each one had a real reason; without writing them down, future-me re-litigates "why did we do it this way." `/create-plan`'s `## Outcomes` section is the right place - it sits right next to the task list, so a future `/review-plan` will see it.

### Trace Annotator v2.1 review bundle (issue #49 / PR #50)

- **Crowded panel? Reroute, don't trim.** When a UI surface drifts beyond its locked spec, "trim or amend the spec" is often a false binary. The third path: re-state what the surface *is for* in one sentence ("decision surface vs tools surface") and route everything that doesn't fit elsewhere instead of cutting features. Honors the original principle, doesn't lose functionality, doesn't require re-litigating the spec. Applied to the right-panel restructure - turned a 9-affordance trim/amend debate into a routing pass.
- **Custom hooks that wrap useState don't inherit the stable-setter exemption from ESLint.** `react-hooks/exhaustive-deps` recognizes `useState`'s setter as stable and lets you omit it from dep arrays. A custom hook returning `[state, setState, ref]` (memoized via `useCallback` with empty deps, identity stable in practice) does NOT get that pass - lint warns on every consumer. Cheapest fix: include the setter in deps explicitly. Zero behavioral cost since identity is stable, and it keeps the lint output honest.
- **`beforeunload` is unreliable; pair with `visibilitychange -> hidden` for mobile.** iOS Safari often skips `beforeunload` entirely; tab switches and backgrounding don't fire it consistently on desktop either. For best-effort flushes (autosave, telemetry), listen on both: `window.beforeunload` for desktop tab close, `document.visibilitychange` checking `document.visibilityState === "hidden"` for mobile and backgrounded states. Neither waits for promises - both are fire-and-forget against IndexedDB.
- **Max-wait debounce is "first pending" time, not "last save" time.** First instinct was to track time-since-last-successful-save and flush immediately when elapsed exceeded the threshold. That breaks for slow-paced labelers: after a long idle stretch, the very first edit writes synchronously, defeating the debounce's purpose. Track when the *first pending change in the current streak* happened instead, and reset only on save success. Now single edits after idle still go through the full debounce; only continuous streaks force the flush.

### Trace Annotator v3.2 (issue #55)

- **A Settings toggle that doesn't change a visible thing is a lie.** v3.2 added a Density preference; the implementation flipped a `density-dense` class on the root, the class only changed a `--pad` token, and nothing in the CSS consumed `--pad`. Build + lint passed cleanly, the toggle was discoverable, but flipping it did literally nothing. `/review` caught it. Pattern: every Settings option must have at least one verifiable visual or behavioural effect; the ship-time check is "name one rule that consumes the thing this toggle changes." If you can't, the toggle is either broken or premature.
- **Don't trust prior "I updated the doc" claims - read the file.** Two earlier conversations claimed the design handoff "now makes three-pane the default in §2." `/explore` opened the file and found the original variant footnote untouched. The user had drafted a blog post celebrating the doc change. Pattern: any time you say "I updated X," your next move should be to read X back, not to rely on memory of the Edit tool firing. Especially across compaction boundaries where state can drift.
- **Solving placement before solving information architecture is wasted work.** Issue #55 started as "where do the displaced top-bar buttons go?" and converged on a kebab menu. Then `/explore` discovered the queue rail was missing - and adding it dissolved most of the placement question (queue absorbs counter, labeled count, jump-to-next-unlabeled, find-as-filter). The kebab still exists but holds half what it would have without the queue. Pattern: when a UI question is "where do these go," check first whether a missing surface should be carrying them.
- **Filter the visible string, not the source field.** v3.2 derived a queue title via `deriveTraceTitle()` (metadata.title -> query -> question -> prompt -> first user message -> id) so the queue showed something readable instead of just an id. The filter still compared `metadata.title ?? trace.id`. Result: users typed what they saw, got zero matches. Pattern: when you transform data for display, the same transform must drive any filter / sort / search that targets the display. WYSIWYF.
- **`role="button"` cannot host an interactive descendant.** Mixed-affordance list rows are common - click the row to navigate, click an inner control (checkbox, three-dot menu) for a side-action. Wrapping the whole row as `role="button"` plus an inner `<input type="checkbox">` is invalid ARIA; the checkbox needs `e.stopPropagation()` to work, and screen readers may swallow the inner control. Pattern: split the row into a presentational grid hosting two interactive siblings (a real `<button>` and a real `<input>`) - never nest interactives.
- **The queue's basic promise is "where am I." Without auto-scroll, that breaks at trace 30.** Keyboard verdict bursts (P/F/U) drove the active-row highlight off-screen within a minute on a 100-trace file. The queue then communicates nothing useful. Pattern: any "where am I" surface must keep the active item visible. `el.scrollIntoView({ block: "nearest" })` is the cheap fix; `nearest` only scrolls when needed so the user isn't yanked around.
- **Audit-log polling that nothing reads is dead code dragging an effect.** v3.2 removed the time-remaining estimator; `auditRecent` and `auditWriteCount` were only feeding it. The audit *writes* still need to happen (per-label history) but the *fetch* effect doesn't. Pattern: when you remove a consumer of derived state, walk back to its inputs - the polling + state + setter often dies with it. ESLint will catch the obvious unused-var, but the dependency cascade (effect, ref bumps, downstream callbacks holding the bumper) hides one level deeper.

### Trace Annotator launch polish (issue #51)

- **Resume a paused plan by refresh, not rewrite.** PLAN-issue-51 paused mid-Step C when first-principles review surfaced a real UX flaw. Three external issues shipped while it was paused (#52, #53, #55). On resumption, the goal-state, audience, and voice all survived intact; only tactical pieces (screenshot spec, three README bullets) had gone stale. Editing the plan in place beat rerunning `/explore` from scratch. Pattern: when external work invalidates a paused plan, ask "did the goal-state change, or just the path?" If only the path, refresh in place. Full rewrites lose the decisions you already made and the rationale captured in earlier Outcomes.
- **Launch prep is its own bug-finding mechanism.** Doing screenshots and first-principles HCI review on a "shipped" UI surfaced #52 (tag input below trace), #58 (bulk action bar below scroll), and #59 (hotkeys swallowed by input focus) - all real bugs that v3.0 / v3.1 / v3.2 feature reviews missed. Pattern: feature reviews check what the feature is supposed to do; launch prep checks what the feature actually does when watched by a stranger holding a stopwatch. Schedule launch prep separately from feature review and treat its bug yield as expected, not surprising.
- **Don't use named UI vocabulary in alt text or marketing copy before defining it.** Initial hero alt text said "queue rail with mixed pass/fail/unlabeled rows on the left, and the decision rail with verdict buttons on the right" - terms not introduced anywhere else in the README until 76 lines later. `/review-copy` flagged it. Pattern: alt text inherits the writer's vocabulary, not the reader's. Test a screen-readable description by stripping all proper nouns from your UI and checking whether what's left orients a newcomer ("a list of traces, a chat in the middle, action buttons on the right" beats "queue rail / decision rail" every time).
- **Plans need a single source of truth for verifiable claims.** Word counts cited at three points across PLAN-issue-51 disagreed (1178, 1383, 1443). All three were "true at the time" but became confusing once the file held all three side by side. `/review-plan` flagged it. Pattern: any number that can be re-derived externally (word count, file size, line count, test count) should be sourced once and referenced by reference, not by value. If multiple checkpoints must record the value over time, label each one with the snapshot context (e.g., "1178 at first write; 1383 after the v3.1 / v3.2 section additions; 1443 final post-review-fixes").
- **`height: 100%` is inert without a parent height chain.** Issue #58 fix added `grid-template-rows: minmax(0, 1fr)` to `.lv-body` - structurally correct, but the action bar still ran off-screen. Cause: `.labeling-view { height: 100% }` resolved to `auto` because nothing in `<html>` / `<body>` / wrapping `<main>` set an explicit height. So `flex: 1` on `.lv-body` had no real upper bound and the "capped" grid row still grew to content. Fix was `height: 100dvh` directly on the labeling root, with a `100vh` fallback. Pattern: when a flex/grid layout that "should work" doesn't, walk the height chain from `<html>` down before reaching for other techniques. Lint and build pass cleanly even when `height: 100%` is silently broken; only an interactive user QA pass surfaces it.
- **Don't run `npm run build` while `npm run dev` is up.** Both write to `.next/` and the dev server starts returning 500s on every request once the build clobbers the manifest files. Recovery requires killing the dev server, deleting `.next/`, and restarting. For a quick type check during dev, run `tsc --noEmit` - it's the type-check half of `next build` without touching `.next/`. Or stop the dev server first. Pattern: shared on-disk caches between watch-mode and one-shot tooling are a footgun; use the build-free type checker for verification when the watcher is live.
